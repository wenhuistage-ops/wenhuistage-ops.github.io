## 🏷️ 文檔分類規則

### **architecture/** - 系統設計文檔
- **適用於**：架構師、高級開發人員、系統設計討論
- **內容**：系統設計、模塊劃分、資料模型、流程圖
- **更新頻率**：設計變更時更新
- **檔名規則**：`[主題].md` 繁體中文（例：`專案架構.md`、`系統流程.md`）

### **guides/** - 開發實踐文檔
- **適用於**：開發人員、新同事入職
- **內容**：環境搭建、部署說明、編碼規範、API 文檔
- **更新頻率**：定期更新（變更開發環境、新增 API 等）
- **檔名規則**：`[功能].md` 繁體中文（例：`開發環境搭建.md`、`編碼規範.md`）

### **issues/** - 問題與改善文檔
- **適用於**：項目經理、QA、開發主管
- **內容**：Bug 列表、安全隱患、性能問題、改進建議
- **更新頻率**：發現問題時更新、每週 review
- **檔名規則**：`[問題類型].md` 繁體中文（例：`安全隱患.md`、`重構計劃.md`）

## 📝 文檔編寫規範

### **通用格式**
```markdown
# [標題]

## 📋 目錄（可選）

## 簡介
[簡潔說明本文檔的用途]

## 主要內容
[分章節組織]

### 代碼示例
[添加代碼時使用適當的語言標簽]

## 相關文檔
- [鏈接到相關文檔] ← 便於導航

---

*最後更新：YYYY年MM月DD日 | 維護人：[名稱]*
```

### **檔名規範**
- 使用繁體中文命名：`專案架構.md` ✅
- 避免英文、特殊符號（除中英數和空格）
- 名稱盡量簡短但足夠清晰（6-20字）

### **內容規範**
- **第一行**必須是 Markdown 標題 `# ...`
- **添加 emoji**提高可視性（但不過過）
- **代碼示例**需要清晰的上下文註解
- **時間敏感內容**要標註更新時間
- **鏈接**優先使用相對路徑

## 🔄 文檔維護流程

### **新增文檔**
1. 確定文檔分類（architecture/guides/issues）
2. 使用規範的繁體中文檔名：`[主題].md`
3. 添加到此份清單中
4. 在 `docs/README.md` 中更新導航

### **更新文檔**
1. 修改相應文檔內容
2. 更新 "最後更新" 時間戳
3. 如有重大變更，更新相關文檔的 "相關文檔" 部分

### **廢棄文檔**
1. 改名為 `_DEPRECATED_XXX.md`
2. 在文檔頂部添加棄用通知
3. 指向新的替代文檔
