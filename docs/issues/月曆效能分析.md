## 🔍 目前問題與根因

### 問題 A：後端仍有進一步優化空間
- 原因：即便已縮小資料量，`getAttendanceRecords()` 仍依賴 `getRange().getValues()` 批次讀取資料
- 需改進：加入後端快取與更精準的範圍讀取策略

### 問題 B：API 資料結構仍可更精簡
- 原因：目前月曆 API 已經簡化，但若仍返回部分不必要欄位，仍會浪費傳輸成本
- 需改進：前端月曆只保留必要字段，後端依圖用途拆分 API 回應

### 問題 C：前端快取管理未加入淘汰策略
- 原因：目前 `monthDataCache` 未限制大小，長期使用可能造成記憶體累積
- 需改進：加入 LRU / 最多保留 12 個月快取策略

// ✅ 改進方式：使用 Query 語言或直接定位範圍
function getAttendanceRecords(monthParam, userIdParam) {
    const sheet = SpreadsheetApp.getActiveSpreadsheet().getSheetByName(SHEET_ATTENDANCE);
    
    // 1. 取得表頭
    const headers = sheet.getRange(1, 1, 1, sheet.getLastColumn()).getValues()[0];
    const dateCol = headers.indexOf('打卡時間') + 1;
    const userIdCol = headers.indexOf('使用者ID') + 1;
    
    // 2. 計算月份範圍（例如：2026-04 → 2026-04-01 to 2026-04-30）
    const [year, month] = monthParam.split('-');
    const startDate = new Date(year, month - 1, 1);
    const endDate = new Date(year, month, 0, 23, 59, 59);
    
    // 3. 使用 filter() 方法快速過濾
    const range = sheet.getDataRange();
    const values = range.getValues().slice(1); // 仍需讀取，但提前知道範圍可優化
    
    return values.filter(row => {
        const date = new Date(row[dateCol - 1]);
        const userId = row[userIdCol - 1];
        
        return date >= startDate && 
               date <= endDate && 
               (!userIdParam || userId === userIdParam);
    }).map(r => ({...}));
}
```

// ✅ 改進版本：簡化返回數據結構
function checkAttendance(attendanceRows) {
    const dailyStatus = []; // 最終返回的數據
    const dailyRecords = {};
    
    // 第一遍：分組
    attendanceRows.forEach(row => {
        const date = getYmdFromRow(row);
        const userId = row.userId;
        const key = `${userId}_${date}`;
        
        if (!dailyRecords[key]) {
            dailyRecords[key] = {
                userId: userId,
                date: date,
                punchInCount: 0,
                punchOutCount: 0,
                hasAdjustment: false,
                approvedAdjustmentCount: 0,
                totalAdjustments: 0,
                records: []
            };
        }
        
        // 同時更新計數器，避免後續重複計算
        if (row.type === "上班") dailyRecords[key].punchInCount++;
        if (row.type === "下班") dailyRecords[key].punchOutCount++;
        if (row.note === "補打卡") {
            dailyRecords[key].hasAdjustment = true;
            dailyRecords[key].totalAdjustments++;
            if (row.audit === "v") dailyRecords[key].approvedAdjustmentCount++;
        }
        
        dailyRecords[key].records.push(row);
    });
    
    // 第二遍：判斷狀態（使用預計算的計數）
    for (const key in dailyRecords) {
        const item = dailyRecords[key];
        
        let reason = "";
        const hasPair = item.punchInCount > 0 && item.punchOutCount > 0;
        const isAllApproved = item.totalAdjustments > 0 && 
                             item.approvedAdjustmentCount === item.totalAdjustments;
        
        if (!hasPair) {
            reason = item.punchInCount === 0 && item.punchOutCount === 0 
                ? "未打上班卡, 未打下班卡" 
                : item.punchInCount > 0 
                ? "未打下班卡" 
                : "未打上班卡";
        } else if (isAllApproved) {
            reason = "補卡通過";
        } else if (item.hasAdjustment) {
            reason = "有補卡(審核中)";
        } else {
            reason = "正常";
        }
        
        // ✅ 簡化返回數據，只包含必要信息
        dailyStatus.push({
            date: item.date,
            userId: item.userId,
            reason: reason,
            hours: calculateWorkHours(item.records), // 計算工時
            isHoliday: checkIfHoliday(item.date) // 判斷假日
        });
    }
    
    return dailyStatus;
}

## 🚀 新增月曆效能規劃

### 1. 後端快取層（高優先級）
- 在 `GS/DbOperations.gs` 使用 `CacheService.getScriptCache()` 快取月曆查詢結果
- 快取鍵可用 `attendance_${month}_${userId || 'all'}`
- 快取時效可設 5~15 分鐘，重複請求命中可大幅減少 Apps Script 執行次數
- 適用於使用者重複瀏覽同一月份或管理員查詢相同員工時

### 2. 資料傳輸精簡（高優先級）
- 月曆專用 API 只返回 `date`, `reason`, `hours`, `isHoliday`
- 不再傳送完整 `record` 列表、`location`、`audit` 等細節
- 前端月曆渲染只需要三項資料即可，API 回傳尺寸可減少 70% 以上
- 已實作 `GS/Handlers.gs` 中的 `handleGetCalendarSummary()`，可進一步做快取與壓縮

### 3. 前端快取最佳化（中優先級）
- 修正 `js/ui.js` 中 `renderCalendar(date, isrefresh)` 的 `monthkey` 產生，確保使用傳入 `date` 而非全域 `currentMonthDate`
- 為月曆資料建立快取淘汰策略，如 `MAX_CACHE_SIZE = 12`
- 只保留最近 12 個月或 3 個月的快取，避免前端記憶體累積

### 4. 預加載策略（中優先級）
- 當用戶載入當前月份後，非阻塞地預加載前一個月與下一個月
- 以 `setTimeout` 延遲 500ms / 1000ms 發起預加載，避免占用主線程和頻寬
- 若使用者有連續向前或向後翻頁行為，可加上簡單行為預測

### 5. DOM 批次更新與虛擬渲染（低優先級）
- `renderCalendarWithData()` 改用 `DocumentFragment` 一次插入所有日期元素
- 若日曆格數過多，可考慮只渲染可見日期範圍，降低 DOM 重繪成本
- 未來若加入更多卡片/詳細內容，可分離「日曆格子渲染」與「點擊詳細顯示」

### 6. 進階選項
- 使用 `Web Worker` 將月曆資料計算移到背景執行緒，避免主執行緒卡頓
- 若部署成靜態前端，可考慮 `Service Worker` 快取月曆 API 請求

function handleGetCalendarSummary(params) {
    const { month, userId } = params;
