### 1. 送出申請後「審核中」徽章永遠不會出現，且異常列表以過期快取重算，員工易重複申請
- **位置**：`js/punch/abnormal-records.js:150`　**維度**：ux-月曆請假　**原則**：狀態回饋 / 資料一致性
- **問題**：兩個缺陷疊加使申請後的狀態回饋完全失效：(1) enrichAbnormalRecordsWithApplicationStatus 以 `app.date || app.displayDate` 做日期比對（abnormal-records.js:150），但後端 getReviewRequest 回傳的項目只有 id/userId/name/type/remark/applicationTime/targetTime/audit/hasProof（firebase-functions/functions/src/getReviewRequest.js:85-96），沒有 date 或 displayDate 欄位，所有申請都被歸到 key 'undefined'，record.status 永遠不會被設成 'pending'，第 253-257 行的「審核中」徽章是死碼。(2) make-up.js 送出成功後先呼叫 refreshAbnormalAfterApplication()（make-up.js:540、560、620）再呼叫 _refreshCalendarAfterMakeup()；checkAbnormal 內 loadMonthDetailData 會同步命中尚未被清除的舊 month 快取（js/ui.js:145-147），把送出前的舊狀態重新算成異常清單並再寫入 abnormal 快取 5 分鐘（abnormal-records.js:121）。結果：員工按下補打卡/請假送出後，異常列表上該日的操作按鈕依然存在、看不到任何審核中提示，誤以為失敗而再送一次；後端 adjustPunch 無查重（adjustPunch.js:42 無條件 add），會產生重複的待審核申請。
- **建議**：後端 getReviewRequest 回傳項目補一個 date 欄位（targetTime 的 YYYY-MM-DD 部分）供前端比對；前端把 make-up.js 中 refreshAbnormalAfterApplication() 與 _refreshCalendarAfterMakeup() 的呼叫順序對調（先清 month 快取再重算異常）。
- **驗證**：三個環節皆經程式碼證實：abnormal-records.js:150 比對的 date/displayDate 欄位在 getReviewRequest.js:85-96 回傳中不存在，「審核中」徽章（253-257行）為死碼；make-up.js:540/560/620/669 先刷異常（同步命中 ui.js 未清的 month 快取並回寫 5 分鐘 abnormal 快取）後才清快取；adjustPunch.js:42 無條件 add 無查重，會產生重複待審核申請。

### 2. adminMonthDataCache 兩種 key 格式並存，編輯/刪除/補卡後月曆顯示舊資料
- **位置**：`js/admin.js:538`　**維度**：ux-管理後台　**原則**：DRY / 快取一致性 / 操作結果回饋
- **問題**：renderAdminCalendar 寫入快取用 `${userId}-${year}-${month}`（admin.js:61），但 loadMonthDetailData（ui.js:139）與 _refreshAdminAfterMutation（admin.js:538）、代補卡 modal（admin.js:865）都用 `${monthKey}-${userId}`。mutation 後刪的是錯的 key，renderAdminCalendar 重繪時命中舊快取：管理員刪除紀錄後看到「紀錄已刪除」通知，但月曆上該紀錄仍在；代員工補卡成功後新卡也不出現，管理員會誤以為操作失敗而重複操作。同時因兩邊 key 對不上，選擇員工時同一份月資料被 getCalendarSummary 抓兩次（月曆一次、詳情/KPI 一次），Firestore reads 直接翻倍。
- **建議**：抽一個 adminMonthCacheKey(userId, monthKey) helper，三處（admin.js:61、538、865 與 ui.js:139、167）統一使用同一格式，讓失效與重用都命中同一個 key。
- **驗證**：屬實。admin.js:61 renderAdminCalendar 快取 key 為 `${userId}-${year}-${monthStr}`（userId-YYYY-MM），但 admin.js:538 與 865 mutation 後刪除的是 `${monthKey}-${userId}`（YYYY-MM-userId），ui.js:139/167 也用後者格式；key 格式對不上導致刪錯 key，重繪時 admin.js:87 命中舊快取顯示過期資料，且同月資料被 getCalendarSummary 重複抓取（月曆與詳情各存一份），Firestore reads 翻倍。line 541 的 cacheManager.invalidate('month') 只清員工自用快取，無法補救。

### 4. 登入按鈕失敗時完全沒有任何回饋，使用者卡在登入頁
- **位置**：`js/app.js:191`　**維度**：ux-錯誤回饋　**原則**：錯誤回饋 / loading 指示
- **問題**：loginBtn.onclick（app.js:188-203）呼叫 callApifetch({action:'getLoginUrl'}) 後只寫 `if (res.url) {...}`，沒有 else、沒有 try/catch、按鈕也沒有 disabled/loading 狀態。正式環境走 Firestore 時 callFirestoreFunction 失敗不會 throw 而是回 {ok:false}，res.url 為 undefined → 點「登入」後畫面毫無反應。Cloud Functions 冷啟動需數秒，期間也無任何指示（預設 loadingId='loading' 的元素不存在，見另一發現），使用者只能反覆狂點或放棄。登入是所有使用者的必經入口，失敗無回饋等同卡死。
- **建議**：在 onclick 加 try/catch 與 else 分支：失敗時 showNotification(t('ERROR_LOGIN_FAILED', {msg: res?.code || ''}))；點擊當下用既有的 generalButtonState(loginBtn, 'processing') 顯示處理中並防連點，結束後恢復。
- **驗證**：屬實。app.js:188-203 的 loginBtn.onclick 僅有 `if (res.url)` 分支，無 else、無 try/catch、無按鈕 disabled；正式環境 config.js 設 useFirestore=true，走 firestore-client.js:79-125 的 callFirestoreFunction，該函式失敗時不 throw 而是回傳 {ok:false,...} 且不呼叫 showNotification，故 res.url 為 undefined 時畫面完全無反應；且 index.html 中確實不存在 id="loading" 元素，冷啟動期間也無任何 loading 指示。

### 5. 匯出月份從「已翻譯的 UI 文字」反解析，越南/英/印尼語系下會靜默匯出錯誤月份
- **位置**：`js/admin.js:2353`　**維度**：原則-單一職責　**原則**：SoC（狀態不應從渲染輸出反解析）、單一資料來源
- **問題**：setupAdminExport（admin.js:2353-2365）與 setupDetailedPayrollExport（admin.js:3155-3167）都用 regex /(\d{4}).*?(\d{1,2})/ 從 adminCurrentMonthDisplay.textContent 反推年月。該文字由 ui.js:228 以 i18n key MONTH_YEAR_TEMPLATE 產生：zh-TW「{year} 年 {month} 月」、ja「{year}年{month}月」可匹配；但 vi「Tháng {month} năm {year}」（月在年前）、en-US 與 id「{month} {year}」（年後無數字）regex 必定匹配失敗。setupAdminExport 失敗時 fallback 到 new Date()（admin.js:2362-2364），管理員在 vi/en/id 介面翻到過去月份按「匯出 Excel」，會拿到「今天所在月份」的打卡資料且無任何提示。index.html:382-383 確認語言選單開放 id/vi 給所有使用者含管理員。權威狀態 adminCurrentDate 明明存在（state.js:46，且 admin.js:1483/1493 月份切換時即時維護），卻繞去解析渲染結果，是 UI 與業務狀態混雜造成錯誤資料的具體案例。
- **建議**：兩個匯出 handler 都直接讀 adminCurrentDate 取 year/month，刪除 monthText regex 解析與 fallback 分支（約 -24 行，零新抽象）。
- **驗證**：核實屬實：admin.js:2357 regex /(\d{4}).*?(\d{1,2})/ 要求 4 位數年份後方還有數字，但 i18n 模板 vi「Tháng {month} năm {year}」、en-US/id「{month} {year}」的年份都在最末，必定匹配失敗；setupAdminExport（admin.js:2362-2364）失敗時 fallback 到 new Date()，管理員在這三種語系翻到過去月份匯出會靜默拿到當前月份資料。唯一修正：setupDetailedPayrollExport（admin.js:3164）的 fallback 是 adminCurrentDate || new Date()，因 adminCurrentDate 有正確維護，該匯出實際不受影響，發現中對此函式的指控過度。

### 6. verifySession 完全不檢查員工 status，未啟用/停用/已離職帳號仍可打卡、請假、補卡
- **位置**：`firebase-functions/functions/src/_helpers.js:84`　**維度**：原則-防禦邊界　**原則**：防禦性程式設計——授權檢查不完整（信任邊界）
- **問題**：verifySession（_helpers.js:84-136）只驗 session 存在、未過期、employees doc 存在，從未檢查 user.status。證據鏈：(1) upsertEmployee（_helpers.js:261）新員工預設 status='未啟用'，但 getProfile.js:113-114 對任何完成 LINE 登入者無條件發 session token；(2) setEmployeeStatus.js 可將員工設為 '停用'/'已離職'，但該員工既有 session（30 天 TTL）與重新 LINE 登入取得的新 session 都照樣通過驗證；(3) punch.js:48、submitLeave.js:45、adjustPunch.js:28 全部只靠 verifySession，後端無任何 status 攔截（grep 確認 status 檢查只存在於排程任務 checkYesterdayPunch/dailyVirtualPunch 與 admin UI）；(4) _helpers.js:62 註解甚至聲稱「status 改未啟用最多 60 秒後生效」——實際上永遠不會生效。結果：已離職員工可持續產生打卡/請假紀錄污染考勤與薪資資料，任意 LINE 帳號（未經管理員啟用）也能直接寫入 attendance 並觸發 LINE 通知轟炸管理員。
- **建議**：在 verifySession 取得 userSnap 後加一段檢查：status 為 '停用' 或 '已離職' 或 '未啟用' 時回傳 { ok:false, code:'ERR_ACCOUNT_INACTIVE' }（空值視為啟用以相容舊資料），並在 i18n 五語系補上對應訊息。既有 60 秒 session cache 機制不用動。
- **驗證**：屬實。verifySession（_helpers.js:84-136）僅檢查 session 存在、未過期、employees doc 存在，回傳結果完全不含 status 判斷；grep 確認 punch.js:48、submitLeave.js:45、adjustPunch.js:28 及 getProfile.js 全無 status 檢查，status 過濾僅存在於排程任務（checkYesterdayPunch.js:164、dailyVirtualPunch.js:238），且 upsertEmployee（_helpers.js:261）預設 status='未啟用' 但仍可正常取得 session 並寫入資料，_helpers.js:62 註解宣稱的 status 生效機制實際不存在。

### 7. 員工端 punch 與 adjustPunch 未驗證 type 白名單，可偽造「請假/休假」類型紀錄
- **位置**：`firebase-functions/functions/src/adjustPunch.js:47`　**維度**：原則-防禦邊界　**原則**：防禦性程式設計——枚舉值白名單驗證缺失
- **問題**：punch.js:111 與 adjustPunch.js:47 都是 `type: type || ""` 原樣寫入，對比同專案 punchWithoutLocation.js:19-21 與 updateAttendanceAsAdmin.js:45-46 都有 ['上班','下班'] 白名單——員工端反而沒有。可利用點：員工直接呼叫 adjustPunch 傳 type='請假'，該筆在 getReviewRequest.js:90 的審核清單顯示 remark='補打卡'（管理員以為在核准一筆補卡），核准後 _attendance.js:257 的 summarizeByDay 以 /請假/.test(type) 判定 → 該日 reason 變 STATUS_LEAVE_APPROVED，月曆與薪資端出現一天「已核准請假」，繞過 submitLeave 的正規請假流程（含 reason 必填與病假證明）。任意垃圾 type 字串也會直接污染 Excel 匯出。

### 8. 請假原因以「翻譯後文字」存入資料庫，跨語言使用者造成統計資料不一致
- **位置**：`js/punch/make-up.js:375`　**維度**：ux-多語系　**原則**：i18n 設計（資料與呈現未分離）→ 錯誤資料
- **問題**：請假表單的 <option> value 直接用翻譯結果：`<option value="${t('LEAVE_SICK') || '病假'}">`（make-up.js:375-377），提交時 reason 原樣送到後端，firebase-functions/functions/src/submitLeave.js:77-78 將其原樣存入 Firestore（`locationName: reason` 與 `reason`）。同一種病假，中文介面存「病假」、日文存「病欠」、印尼文存「Izin Sakit」、越南文（目前是英文）存「Sick Leave」。js/admin.js:3656-3668 的請假統計以此原始字串當分組 key（`leaveStats[category]`），同類假別會被拆成多個類別，管理員月報表統計數字錯誤，且管理員介面會直接看到外文假別名稱。

- **驗證**：屬實。make-up.js:375-377 的 option value 直接使用 t('LEAVE_SICK') 等翻譯字串，i18n 各語系值不同（zh-TW「病假」、ja「病欠」、id「Izin Sakit」、en/vi「Sick Leave」），submitLeave.js:77-78 將 reason 原樣寫入 locationName 與 reason，admin.js:3662 以 leaveRec.location 原始字串當統計分組 key，同一假別會依使用者語系被拆成多個類別，統計與報表分類確實不一致。
