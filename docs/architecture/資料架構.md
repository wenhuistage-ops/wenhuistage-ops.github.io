# 資料架構分析

## 📊 資料存儲概述

本系統採用 **Google Sheets** 作為唯一的資料持久化層。所有數據通過 Google Apps Script 進行讀寫操作，不涉及傳統數據庫。

---

## 🗂️ Google Sheets 表單結構

### **1. EMPLOYEES（員工表）**

| 欄位 | 類型 | 說明 | 範例 |
|------|------|------|------|
| A: userId | String | LINE 用戶 ID（主鍵） | `U1234567890abcdef` |
| B: email | String | 員工電子郵件 | `employee@company.com` |
| C: displayName | String | 員工名稱 | `李明` |
| D: pictureUrl | String | LINE 頭像 URL | `https://...jpg` |
| E: createdAt | DateTime | 記錄創建時間 | `2025-01-15 10:30:00` |
| F: department | String | 部門（選項式） | `管理員` / `技術部` / `營運部` |
| G: *reserved* | String | 預留欄位 | - |
| H: status | String | 狀態 | `啟用` / `未啟用` / `已離職` |

**主要功能**：
- 首次登入時自動建立記錄
- 存儲用戶基本信息＆權限狀態
- `status` 欄位用於控制帳戶啟用/禁用

**查詢操作**：
```javascript
findEmployeeByLineUserId_(userId)  // 根據 userId 查找員工
writeEmployee_(profile)             // 首次登入時建立員工記錄
```

### **2. SESSION（會話表）**

| 欄位 | 類型 | 說明 | 範例 |
|------|------|------|------|
| A: sessionToken | String | Session 令牌（主鍵） | `uuid-uuid-uuid-uuid` |
| B: userId | String | 員工 ID（外鍵） | `U1234567890abcdef` |
| C: createdAt | DateTime | Token 建立時間 | `2025-01-15 10:30:00` |
| D: expiredAt | DateTime | Token 過期時間 | `2025-01-15 12:30:00` |

**主要功能**：
- 管理用戶登入會話
- 實現 Token 安全驗證機制
- Session TTL（生存時間）預設為 2 小時

**操作流程**：
```javascript
writeSession_(userId)        // 生成新 Session
verifyOneTimeToken_(otoken)  // 兌換一次性令牌為 sessionToken
checkSession_(sessionToken)  // 驗證 Session 有效性
```

**流程圖**：
```
用戶授權 → 生成 oneTimeToken → Token 發送到前端
  ↓
前端調用 getProfile → 兌換為 sessionToken
  ↓
sessionToken 存入 localStorage
  ↓
後續每次 API 調用都驗證 sessionToken 有效性
```

### **3. PUNCH_RECORDS（打卡記錄表）**

| 欄位 | 類型 | 說明 | 範例 |
|------|------|------|------|
| A: recordId | String | 記錄 ID（主鍵） | `uuid-1234...` |
| B: userId | String | 員工 ID（外鍵） | `U1234567890abcdef` |
| C: recordDate | Date | 打卡日期 | `2025-01-15` |
| D: punchInTime | DateTime | 上班時間 | `2025-01-15 09:00:30` |
| E: punchOutTime | DateTime | 下班時間 | `2025-01-15 18:15:45` |
| F: punchInLocation | String | 上班地點 | `辦公室` |
| G: punchOutLocation | String | 下班地點 | `辦公室` |
| H: punchInCoordinates | String | 上班 GPS 坐標 | `25.033,121.543` |
| I: punchOutCoordinates | String | 下班 GPS 坐標 | `25.033,121.543` |
| J: status | String | 狀態 | `已驗證` / `待審核` / `異常` |
| K: notes | String | 備註 | `補卡申請：會議延誤` |
| L: createdAt | DateTime | 記錄建立時間 | `2025-01-15 10:30:00` |
| M: updatedAt | DateTime | 最後更新時間 | `2025-01-16 14:20:00` |

**主要功能**：
- 記錄每次打卡事件（上班/下班）
- 存儲 GPS 坐標用於位置驗證
- 追蹤記錄的狀態變更

**狀態流轉**：
```
立即打卡 → 已驗證（自動）
補卡申請 → 待審核（管理員審核）→ 已驗證 / 駁回
異常記錄 → 異常（檢查邏輯觸發）
```

### **4. LOCATIONS（打卡地點表）**

| 欄位 | 類型 | 說明 | 範例 |
|------|------|------|------|
| A: locationId | String | 地點 ID（主鍵） | `LOC001` |
| B: locationName | String | 地點名稱 | `台北總公司` |
| C: latitude | Number | 緯度 | `25.033` |
| D: longitude | Number | 經度 | `121.543` |
| E: radius | Number | 允許打卡半徑（公尺） | `100` |
| F: description | String | 地點描述 | `台北市大安區` |
| G: isActive | Boolean | 是否啟用 | `TRUE` / `FALSE` |
| H: createdAt | DateTime | 建立時間 | `2025-01-01 10:00:00` |
| I: createdBy | String | 建立人（Admin ID） | `U1234567890abcdef` |

**主要功能**：
- 定義可打卡的地點及其地理範圍
- 支持多地點打卡設置
- 管理員透過 UI 新增/修改地點

**驗證邏輯**：
```javascript
// 計算用戶位置與定義地點的距離
function isWithinRadius_(userLat, userLng, locationLat, locationLng, radius) {
  const distance = calculateDistance(userLat, userLng, locationLat, locationLng);
  return distance <= radius;
}
```

| 欄位 | 類型 | 說明 | 範例 |
|------|------|------|------|
| A: requestId | String | 申請 ID（主鍵） | `REQ001` |
| B: userId | String | 申請人 ID（外鍵） | `U1234567890abcdef` |
| C: requestDate | Date | 補卡日期 | `2025-01-14` |
| D: requestTime | DateTime | 補卡時間 | `2025-01-14 09:30:00` |
| E: punchType | String | 打卡類型 | `上班` / `下班` |
| F: reason | String | 補卡原因 | `会議延誤，10分鐘後打卡` |
| G: status | String | 申請狀態 | `待審核` / `已批准` / `已駁回` |
| H: reviewedBy | String | 審核人 ID | `U9876543210fedcba` |
| I: reviewedAt | DateTime | 審核時間 | `2025-01-15 10:30:00` |
| J: createdAt | DateTime | 申請建立時間 | `2025-01-15 09:00:00` |

**主要功能**：
- 員工提交補卡申請

- 批准後寫入 PUNCH_RECORDS 表

**工作流程**：
```
員工提交補卡 → 待審核
                ↓
            管理員審核
                ├─ 批准 → 寫入打卡記錄 → 已批准
                └─ 駁回 → 已駁回（不寫入記錄）
```

### **6. ANOMALY_RECORDS（異常記錄表）**

| 欄位 | 類型 | 說明 | 範例 |
|------|------|------|------|
| A: anomalyId | String | 異常記錄 ID（主鍵） | `ANO001` |
| B: userId | String | 員工 ID（外鍵） | `U1234567890abcdef` |
| C: recordDate | Date | 異常日期 | `2025-01-15` |
| D: anomalyType | String | 異常類型 | `未打上班` / `未打下班` / `重複打卡` |
| E: description | String | 異常描述 | `該日期無上班記錄` |
| F: severity | String | 嚴重程度 | `高` / `中` / `低` |
| G: status | String | 處理狀態 | `未處理` / `已審視` / `已處理` |
| H: detectedAt | DateTime | 檢測時間 | `2025-01-16 10:00:00` |

**主要功能**：
- 自動檢測打卡異常（缺卡、重複卡等）
- 提醒（警告）管理員和員工

**異常檢測邏輯**：
```javascript
function checkAbnormal(userId, date) {
  const records = getPunchRecordsForDate(userId, date);
  
  // 檢查1：未打昨班卡
  if (!records.punchIn) {
    return { type: "未打上班", severity: "高" };
  }
  
  // 檢查2：未打下班卡
  if (!records.punchOut) {
    return { type: "未打下班", severity: "中" };
  }
  
  // 檢查3：時間邏輯異常
  if (records.punchIn > records.punchOut) {
    return { type: "時間異常", severity: "高" };
  }
}
```

## 🔄 資料流程

### **流程 1：首次登入**
```
1. 用戶點擊 LINE Login
2. LINE API 返回用戶資訊（userId, name, picture）
3. GAS 呼叫 writeEmployee_() → 檢查 EMPLOYEES 表
4. 若不存在，則新增一行
5. 返回員工資訊給前端
```

### **流程 2：打卡操作**
```
1. 前端獲取 GPS 位置（lat, lng）
2. 呼叫 /punch API，傳送位置＆時間戳
3. GAS 驗證：
   - Session Token 有效？
   - 位置在允許範圍內？
   - 時間邏輯是否正確？（上班不能晚於下班）
4. 若驗證通過，插入 PUNCH_RECORDS
5. 若驗證失敗，設置 status 為 "異常"
6. 檢查是否有異常，插入 ANOMALY_RECORDS
```

### **流程 3：補卡申請與審核**
```
1. 員工選擇日期＆時間，填寫原因
2. 提交 /submitMakePunch API
3. GAS 插入 MAKE_PUNCH，status 為 "待審核"
4. 管理員在 Admin Panel 查看待審核申請
5. 點擊 "批准"：
   - 插入 PUNCH_RECORDS 對應記錄
   - 更新 MAKE_PUNCH status 為 "已批准"
6. 或點擊 "駁回"：
   - 更新 MAKE_PUNCH status 為 "已駁回"
```

## 📈 資料量估算

假設公司 100 人，日均出勤 80 人，保留 1 年數據：

| 表單 | 預計行數 | 備註 |
|------|--------|------|
| EMPLOYEES | 100+ | 包含離職員工 |
| SESSION | 500-1000 | 每個活躍會話1行（舊會話刪除） |
| PUNCH_RECORDS | ~29,200 | 80 人 × 365 天 |
| LOCATIONS | 5-20 | 多地點辦公 |
| MAKE_PUNCH | 500-2000 | 年均補卡次數 |
| ANOMALY_RECORDS | 300-1000 | 異常檢測結果 |

**建議**：
- 定期清理過期的 SESSION 記錄（TTL > 2 小時）
- 定期歸檔舊年份的 PUNCH_RECORDS
- 使用 Google Sheets API 批量導出用於備份

## 🔐 資料安全考慮

### **現有措施**
- Session Token 有 TTL 限制（預設 2 小時）
- 每次操作驗證 sessionToken
- localStorage 中不存儲敏感信息

### **建議改進**
- ✅ 啟用 Google Sheets 行級別訪問控制
- ✅ 定期備份 Google Sheets（自動導出＆雲存儲）
- ✅ 加密敏感欄位（如座標）
- ✅ 實現完整的審計日誌（記錄所有修改操作）
- ✅ 定期輪換 LINE Channel Secret
