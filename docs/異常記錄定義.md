# 打卡系統異常記錄定義

## 概述

打卡系統會自動檢查員工的每日打卡記錄，並識別出異常情況。這些異常記錄會顯示在用戶介面中，讓員工可以進行補打卡操作。

## 異常記錄類型

### 核心異常情況（用戶介面可見）

#### 1. STATUS_BOTH_MISSING (本日未打卡)
- **定義**: 當天上班和下班打卡都缺少
- **觸發條件**: `punchInCount === 0 && punchOutCount === 0`
- **顯示文字**: "本日未打卡"
- **前端按鈕**: 顯示「補打卡」、「請假」、「休假」3個按鈕
- **UI 備註**: 補打卡時需填寫上班和下班時間

#### 2. STATUS_PUNCH_IN_MISSING (未打上班卡)
- **定義**: 當天只有下班打卡記錄，缺少上班打卡
- **觸發條件**: `punchInCount === 0 && punchOutCount > 0`
- **顯示文字**: "未打上班卡"
- **前端按鈕**: 只顯示「補打卡」按鈕
- **UI 備註**: 補打卡時只需填寫上班時間

#### 3. STATUS_PUNCH_OUT_MISSING (未打下班卡)
- **定義**: 當天只有上班打卡記錄，缺少下班打卡
- **觸發條件**: `punchInCount > 0 && punchOutCount === 0`
- **顯示文字**: "未打下班卡"
- **前端按鈕**: 只顯示「補打卡」按鈕
- **UI 備註**: 補打卡時只需填寫下班時間

### 其他狀態（系統內部使用，通常不顯示）

### 4. STATUS_PUNCH_NORMAL (打卡正常)
- **定義**: 當天既有上班打卡又有下班打卡，且沒有補打卡記錄
- **觸發條件**: `punchInCount > 0 && punchOutCount > 0 && !hasAdjustment`
- **顯示文字**: "打卡正常"

### 5. STATUS_REPAIR_PENDING (補卡審核中)
- **定義**: 當天有補打卡記錄，但尚未全部通過審核
- **觸發條件**: `hasAdjustment && approvedAdjustmentCount < totalAdjustments`
- **顯示文字**: "補卡(審核中)"

### 6. STATUS_REPAIR_APPROVED (補卡通過)
- **定義**: 當天所有補打卡記錄都已通過審核
- **觸發條件**: `totalAdjustments > 0 && approvedAdjustmentCount === totalAdjustments`
- **顯示文字**: "補卡通過"

## 時間定義說明

### 申請相關時間
- **申請時間**: 員工提交表單時的系統時間，表示申請何時送出
- **請假/休假時間**: 員工選擇的具體請假日期，表示實際請假的日期
- **補打卡時間**: 員工選擇的補打卡日期時間，表示補打卡的目標時間

### 管理員介面顯示
管理員在審核申請時會看到兩個時間：
- **申請時間**: 顯示在申請詳情中，表示員工何時提交申請
- **請假/補卡時間**: 顯示在申請詳情中，表示申請針對的具體時間

### 資料儲存
- 申請時間儲存在設備信息欄位中，格式為 `申請時間: yyyy-MM-dd HH:mm`
- 請假/補卡時間儲存在打卡時間欄位中

### 資料來源
- 從打卡記錄表中獲取員工的每日打卡資料
- 過濾掉系統虛擬卡記錄 (`note !== "系統虛擬卡"`)

### 處理流程
1. **分組記錄**: 按員工ID和日期分組打卡記錄
2. **預計算計數器**:
   - `punchInCount`: 上班打卡次數
   - `punchOutCount`: 下班打卡次數
   - `totalAdjustments`: 補打卡總數
   - `approvedAdjustmentCount`: 已審核通過的補打卡數
3. **判斷狀態**: 根據計數器結果判斷當日狀態
4. **生成異常記錄**: 對於非正常狀態的記錄，生成異常記錄項目

### 排除條件
- **今日記錄**: 系統不會檢查當天的打卡記錄 (`date === today`)
- **系統虛擬卡**: 過濾掉標記為"系統虛擬卡"的記錄
- **檢查範圍**: 只檢查最近的當月異常記錄（`monthsToCheck = 1`）

## 資料結構

### 前端按鈕邏輯

| 異常類型 | 補打卡 | 請假 | 休假 | 備註 |
|---------|-------|------|------|------|
| STATUS_BOTH_MISSING | ✓ | ✓ | ✓ | 上班下班都沒有時的全功能按鈕 |
| STATUS_PUNCH_IN_MISSING | ✓ | ✗ | ✗ | 只缺上班卡 |
| STATUS_PUNCH_OUT_MISSING | ✓ | ✗ | ✗ | 只缺下班卡 |

### 異常記錄格式
```javascript
{
  date: "2024-01-15",        // 日期 (YYYY-MM-DD)
  reason: "STATUS_BOTH_MISSING",  // 異常原因代碼
  id: "abnormal-1"           // 唯一識別碼
}
```

### 完整狀態記錄格式
```javascript
{
  ok: false,                 // 是否正常 (true: 正常, false: 異常)
  date: "2024-01-15",        // 日期
  record: [                  // 當日打卡記錄詳情
    {
      time: "09:00",
      type: "上班",
      note: "",
      audit: "",
      location: "公司大樓"
    }
  ],
  reason: "STATUS_PUNCH_OUT_MISSING",  // 狀態代碼
  hours: 8.5,              // 工作時數
  id: "abnormal-1"          // 唯一識別碼
}
```

## 前端處理

### 顯示邏輯
- 異常記錄會顯示在"異常記錄"區塊中
- 每個異常記錄提供"補打卡"按鈕
- 根據異常類型隱藏相應的補打卡按鈕：
  - 缺少上班卡時，隱藏補上班卡按鈕
  - 缺少下班卡時，隱藏補下班卡按鈕

### 補打卡流程
1. 用戶點擊"補打卡"按鈕
2. 顯示補打卡表單，選擇日期時間
3. 提交補打卡請求
4. 等待管理員審核

## 後端實作

### 相關函數
- `checkAttendanceAbnormal()`: 檢查異常記錄 (簡化版)
- `checkAttendance()`: 完整狀態檢查 (包含工時計算)
- `handleGetAbnormalRecords()`: API處理器

### 資料來源
- Google Sheets 中的打卡記錄表
- 記錄欄位包含：打卡時間、員工ID、上下班類型、GPS位置等

## 多語言支援

系統支援多語言顯示異常狀態：

| 語言 | 未打上班卡 | 未打下班卡 | 打卡正常 |
|------|-----------|-----------|---------|
| 繁體中文 | 未打上班卡 | 未打下班卡 | 打卡正常 |
| 英文 | Missed Punch In | Missed Punch Out | Normal |
| 日文 | 出勤打刻なし | 退勤打刻なし | 正常 |
| 印尼文 | Tidak absen masuk | Tidak absen keluar | Normal |
| 越南文 | Thiếu chấm công vào | Thiếu chấm công ra | Bình thường |

## 注意事項

1. **即時性**: 異常記錄檢查不包含當天記錄，避免顯示不完整的資料
2. **審核流程**: 補打卡需要管理員審核通過後才會更新狀態
3. **資料完整性**: 系統會過濾無效記錄，確保判斷準確性
4. **效能考量**: 使用預計算計數器優化判斷邏輯，提高處理效率
